# CredPay — Kubernetes Manifests

Plain Kubernetes manifests that deploy the **CredPay** application onto the AKS
cluster provisioned by the `terraform/` project. Everything runs in the
`credpay` namespace and is exposed through a single ingress-nginx entry point.

No Helm, no Kustomize, no GitOps, no service mesh — just `kubectl apply`,
driven either by hand or by [`azure-pipelines.yml`](../azure-pipelines.yml).
This is a classroom capstone, kept intentionally simple.

> **Automated deployment:** `azure-pipelines.yml` at the repo root now runs
> Terraform → Docker build/push → AKS deploy end-to-end on every push to
> `main`, including a blue/green rollout for the frontend. This document
> still describes every manifest and the equivalent manual `kubectl`
> commands — they are exactly what the pipeline runs under the hood.

## How this fits with Terraform

**Terraform (the `terraform/` project) creates the Azure infrastructure:**

- Resource Group
- Virtual Network
- **AKS** cluster
- **Azure PostgreSQL** Flexible Server
- **Log Analytics** Workspace
- PostgreSQL secrets pushed into an **existing** Azure Key Vault

**Kubernetes (this project) deploys the application onto that AKS cluster:**

- Namespace
- ConfigMap
- Secret (recreated from Key Vault on every pipeline run)
- PostgreSQL schema Job
- Deployments (frontend **blue/green**, user-service, payment-service)
- Services
- Ingress

**Container images** are pulled from the **existing** Azure Container Registry
`credproj.azurecr.io` (it is *not* created by Terraform). AKS must be attached
to it once, out-of-band:

```bash
az aks update --attach-acr credproj
```

That grants the AKS kubelet identity `AcrPull`, so **no `imagePullSecrets` are
needed**. There is still **no Workload Identity and no Managed Identity** — the
app authenticates to PostgreSQL with a username + password. The only change
from Phase 1 is *where that password now lives before it becomes a Kubernetes
Secret*: Terraform writes it into an Azure Key Vault, and the pipeline reads it
back out with the `AzureKeyVault@2` task. There is still **no CSI driver** —
pods never talk to Key Vault directly.

## Directory layout

| Path | Kind | Purpose |
|------|------|---------|
| `namespace/namespace.yaml` | Namespace | `credpay` namespace (Pod Security = `baseline`) |
| `configmap/configmap.yaml` | ConfigMap | Non-secret DB coordinates + Spring datasource URL (`credpay-config`) |
| `secrets/secret.yaml` | Secret | **Example only** — the real one is generated by the pipeline from Key Vault |
| `postgres/schema-init-job.yaml` | Job | One-shot job that applies `schema.sql` to PostgreSQL before backends start |
| `frontend/deployment-blue.yaml` | Deployment | Frontend, "blue" color |
| `frontend/deployment-green.yaml` | Deployment | Frontend, "green" color |
| `frontend/hpa-blue.yaml` / `hpa-green.yaml` | HPA | One per color, same CPU-70% target as before |
| `frontend/service.yaml` | Service | Single stable ClusterIP; `selector.version` picks blue or green |
| `user-service/` | Deployment, Service, HPA | Spring Boot / Java 21 REST API on `:8080` (rolling update) |
| `payment-service/` | Deployment, Service, HPA | FastAPI / Python 3 REST API on `:8000` (rolling update) |
| `ingress/ingress.yaml` | Ingress | Host-less, path-based routing to all three services |

## Prerequisites

- Terraform applied, and `kubectl` pointed at the AKS cluster:
  ```bash
  az aks get-credentials \
    --resource-group $(terraform -chdir=../terraform output -raw resource_group_name) \
    --name $(terraform -chdir=../terraform output -raw aks_cluster_name) \
    --overwrite-existing
  ```
- AKS attached to the ACR: `az aks update --attach-acr credproj`
- An Azure Key Vault created out-of-band, with Terraform's identity granted
  permission to set secrets on it (see `terraform/README.md`).
- **ingress-nginx** controller installed in the cluster.
- **metrics-server** running (built into AKS) — required by the HPAs.
- Images pushed to `credproj.azurecr.io`:
  `credpay/frontend`, `credpay/user-service`, `credpay/payment-service`.

## Hardcoded values (no placeholders left)

Both the ACR login server and the PostgreSQL host are already filled in with
real values, so the manifests can be applied as-is:

| Value | Found in | Current value |
|-------|----------|----------------|
| ACR login server | `frontend/`, `user-service/`, `payment-service/` Deployments | `credproj.azurecr.io` |
| PostgreSQL FQDN | `configmap/configmap.yaml` | `psql-credpay.postgres.database.azure.com` |

If either Azure resource is ever renamed, re-run
`terraform -chdir=../terraform output -raw postgres_fqdn` (or the ACR name)
and update these files directly - there is no templating step.

## Secret

`secrets/secret.yaml` is an **example** with `REPLACE_ME` values — never commit a
real password. The pipeline creates the real Secret on every run by reading
the password out of Azure Key Vault and running:

```bash
kubectl create secret generic credpay-db \
  --namespace credpay \
  --from-literal=DB_PASSWORD="$DB_PASSWORD" \
  --from-literal=SPRING_DATASOURCE_PASSWORD="$DB_PASSWORD" \
  --dry-run=client -o yaml | kubectl apply -f -
```

Doing the same thing manually:

```bash
kubectl create secret generic credpay-db \
  --namespace credpay \
  --from-literal=DB_PASSWORD="$(terraform -chdir=../terraform output -raw postgres_admin_password)" \
  --from-literal=SPRING_DATASOURCE_PASSWORD="$(terraform -chdir=../terraform output -raw postgres_admin_password)"
```

This is still a plain Kubernetes Secret — no CSI driver, no Workload Identity.
Key Vault is only ever read by the pipeline agent, never by a pod.

## Deployment order

Apply in this order — config and schema must exist before the backends start:

```bash
# 1. Namespace
kubectl apply -f namespace/namespace.yaml

# 2. ConfigMap  (already has the real Postgres FQDN filled in)
kubectl apply -f configmap/configmap.yaml

# 3. Secret  (create from Key Vault - see "Secret" above)

# 4. Schema Job — first create the schema ConfigMap from the repo's schema.sql:
kubectl create configmap db-schema \
  --namespace credpay \
  --from-file=schema.sql=../schema.sql \
  --dry-run=client -o yaml | kubectl apply -f -
kubectl delete job db-schema-init -n credpay --ignore-not-found
kubectl apply -f postgres/schema-init-job.yaml
kubectl wait --for=condition=complete job/db-schema-init -n credpay --timeout=120s

# 5. User Service (rolling update)
kubectl apply -f user-service/
kubectl rollout restart deployment/user-service -n credpay
kubectl rollout status deployment/user-service -n credpay

# 6. Payment Service (rolling update)
kubectl apply -f payment-service/
kubectl rollout restart deployment/payment-service -n credpay
kubectl rollout status deployment/payment-service -n credpay

# 7. Frontend (blue/green - see below)

# 8. Ingress
kubectl apply -f ingress/ingress.yaml
```

> The `user-service` runs with `spring.jpa.hibernate.ddl-auto=validate`, so the
> tables **must** already exist — that is why the schema Job (step 4) runs before
> the backends.

## Frontend blue/green

`user-service` and `payment-service` still use a plain `RollingUpdate` — only
the **frontend** does blue/green, since it's the one component a broken deploy
of would be immediately visible to every user.

Both `frontend-blue` and `frontend-green` Deployments always exist; the
`frontend` Service's `selector.version` decides which one is actually live.
The pipeline (and the manual steps below) always deploy the fresh image to
whichever color is **not** currently live, smoke-test it directly, and only
flip the Service selector if that passes:

```bash
# 0. Bootstrap ONLY - apply this once, the very first time the Service
#    doesn't exist yet. Do NOT re-run this on every deploy (see warning below).
kubectl get svc frontend -n credpay >/dev/null 2>&1 || kubectl apply -f frontend/service.yaml

kubectl apply -f frontend/deployment-blue.yaml
kubectl apply -f frontend/deployment-green.yaml
kubectl apply -f frontend/hpa-blue.yaml
kubectl apply -f frontend/hpa-green.yaml

# 1. Which color is live right now?
CURRENT=$(kubectl get svc frontend -n credpay -o jsonpath='{.spec.selector.version}')
NEW=$([ "$CURRENT" = "blue" ] && echo green || echo blue)
echo "Live: $CURRENT -> deploying: $NEW"

# 2. Roll the new image out to the idle color only
kubectl rollout restart deployment/frontend-$NEW -n credpay
kubectl rollout status deployment/frontend-$NEW -n credpay --timeout=180s

# 3. Smoke test the idle color BEFORE it takes traffic (see "Smoke test" below)

# 4. Only if the smoke test passed, flip live traffic to it
kubectl patch svc frontend -n credpay --type merge \
  -p "{\"spec\":{\"selector\":{\"app.kubernetes.io/name\":\"frontend\",\"version\":\"$NEW\"}}}"
```

If the smoke test fails, skip step 4 — `$CURRENT` keeps serving traffic and the
broken `$NEW` Deployment is left running (not deleted) so it can be inspected
with `kubectl logs deployment/frontend-$NEW -n credpay`.

> **Do not `kubectl apply -f frontend/service.yaml` on every deploy.** It was
> confirmed live on this cluster that re-applying it resets
> `spec.selector.version` back to the file's hardcoded `blue` default, even
> though the file itself never changed - which silently breaks the whole
> rotation (every run would see "blue" and redeploy to "green" again). Once
> the Service exists, its selector is owned exclusively by the `kubectl patch`
> in step 4 above - see `STAGE2-CHANGES.md` §2.5 for how this was found.

## Health checks & smoke test

Every rollout is followed by `kubectl rollout status` (fails the pipeline if
the Deployment doesn't reach Ready within the timeout) and a
`kubectl get pods` dump. Before the frontend flip specifically, the pipeline
also runs a smoke test **from inside the new color's own pod** — that way it
can reach the in-cluster Service DNS names without needing the public Ingress
IP, and a broken backend is caught before any real user sees it:

```bash
NEW_POD=$(kubectl get pods -n credpay -l app.kubernetes.io/name=frontend,version=$NEW \
  -o jsonpath='{.items[0].metadata.name}')

# 127.0.0.1, not "localhost" - the pod's /etc/hosts resolves localhost to
# ::1 first, but nginx only listens on 0.0.0.0:80 (IPv4), so wget reports a
# false "Connection refused" against "localhost" even when the app is fine.
kubectl exec -n credpay "$NEW_POD" -- wget -q -O /dev/null http://127.0.0.1:80/
kubectl exec -n credpay "$NEW_POD" -- wget -q -O /dev/null \
  http://user-service.credpay.svc.cluster.local:8080/api/users/health
kubectl exec -n credpay "$NEW_POD" -- wget -q -O /dev/null \
  http://payment-service.credpay.svc.cluster.local:8000/api/payment/health
```

`wget`'s own exit code is 0 only on an HTTP 2xx response, so each line above
simply needs to succeed - no output-parsing required. `GET /api/users/health` and
`GET /api/payment/health` are tiny additive endpoints on top of the existing
controllers, added specifically so this smoke test (and the same paths behind
the public Ingress) have something real to call.

## Ingress routing

The Ingress has **no host** — it answers on the ingress controller's public
LoadBalancer IP over HTTP. (Add a DNS name + TLS in a later phase.)

| Path | Backend | Port |
|------|---------|------|
| `/api/payment` | payment-service | 8000 |
| `/api/users` | user-service | 8080 |
| `/api/cards` | user-service | 8080 |
| `/` (everything else) | frontend | 80 |

Because all traffic is same-origin behind the ingress, the frontend uses relative
`/api/...` calls and **no CORS is required**. Build the frontend with empty
`VITE_USER_API_URL` / `VITE_PAYMENT_API_URL`.

## Autoscaling (HPA)

All workloads scale on CPU (user-service also on memory), `min 2 / max 6`,
with a 300s scale-down stabilization window. Requires metrics-server. The
frontend has one HPA per color, each with the same target — only the color
currently receiving traffic will actually see load and scale up.

| Service | Target |
|---------|--------|
| frontend-blue / frontend-green | CPU 70% |
| user-service | CPU 75% + memory 80% |
| payment-service | CPU 75% |

```bash
kubectl get pods -n credpay -o wide
kubectl get svc -n credpay
kubectl get ingress -n credpay
kubectl get deployments -n credpay
kubectl get hpa -n credpay

# Which color is live, and what image is it running?
kubectl get svc frontend -n credpay -o jsonpath='{.spec.selector}'
kubectl get deployments -n credpay \
  -o jsonpath='{range .items[*]}{.metadata.name}{"\t"}{.spec.template.spec.containers[0].image}{"\n"}{end}'

# End-to-end, from outside the cluster
INGRESS_IP=$(kubectl get svc -n ingress-nginx ingress-nginx-controller \
  -o jsonpath='{.status.loadBalancer.ingress[0].ip}')
curl http://$INGRESS_IP/
curl http://$INGRESS_IP/api/users/health
curl http://$INGRESS_IP/api/payment/health
```

Then browse to `http://<INGRESS_IP>/`.

## Notes / current limitations

- **No Workload Identity / Managed Identity, no CSI driver.** Key Vault is
  read only by the pipeline agent (`AzureKeyVault@2`); pods still get the DB
  password purely from a Kubernetes Secret.
- **Frontend runs as root** (`runAsNonRoot: false`) because `nginx:stable-alpine`
  binds `:80`, so namespace Pod Security is `baseline`. Backends run non-root
  with a read-only root filesystem and a writable `/tmp` `emptyDir`.
- **Image pull** relies on `az aks update --attach-acr credproj`; there are no
  `imagePullSecrets`.
- **`imagePullPolicy: Always`** on all app Deployments. All are pushed to the
  mutable `:latest` tag, so `IfNotPresent` would let a node keep running a
  stale cached image after `kubectl rollout restart` - `Always` guarantees
  the freshly-pushed image is pulled every time.
- The schema Job is **safe to re-run, non-destructively** — `schema.sql`
  uses `CREATE TABLE IF NOT EXISTS` and only seeds sample rows into an
  empty table, so real users/cards/payments created through the live app
  survive every deploy.
- Both frontend colors run continuously (nothing is scaled to zero or
  deleted) so a rollback is always just a Service-selector flip away, not a
  redeploy.
